# System Design避坑指南

1. 在没有与面试官沟通或了解系统要求和目标，并未进行容量估算的情况下，仅凭图表或立即建议使用哪种数据库，可能会带来负面影响。另一方面，花费过多时间（超过10分钟）在此部分上，并提供过于详细的计算，可能会导致剩余讨论点的时间不足。
2. 提到特定技术/组件，但缺乏其细节方面的了解。
    1. 例如，您可能会提到需要使用消息队列，但如果您特别提到Kafka/SQS，并被问及使用Kafka的优点、消费者/生产者端点的工作方式、如何确保数据传递和排序等方面，但无法连贯地回答或给出错误的回答，将会被扣分。
    2. 然而，对于高级或更高级别的职位来说，在面试中展示深度也很重要，因此最好对常用技术有透彻的了解。
3. 忽视或忽略面试官的引导，过度讲话或故意炫耀。
    1. 请记住，面试官通常有一些他们想要您讨论的话题，即使是同一个问题，焦点也可能大不相同。因此，让面试官有机会提出他们的问题非常重要。在回答问题之前，确保您了解面试官想要问什么。例如，如果面试官想讨论隔离级别，但您一直在谈论锁，或者如果他们希望您考虑可用性和一致性之间的权衡，但您一直在谈论添加缓存等。
    2. 故意炫耀您的技能，例如与面试官讨论共识算法，而问题场景仅需要一个单机，而且您甚至还没有将其设计为分布式，这是不可取的。此外，即使您对Cassandra有深入了解，这也并不意味着您应该将其用于每个与数据库相关的问题，Redis也是一样。如果在现场面试期间有多个设计环节，这很可能会在总结时被发现。
    3. 设计过于复杂，无论是否需要异步，都在任何两个服务之间添加消息队列，在任何DB之前添加缓存层，无论其是否以读为主。
4. 数据库选择似乎是随意的，或者试图使用一种类型的数据库回答所有问题。即使你说你用 Spanner 来做所有事情，它并不是不能用，但你至少应该解释如何在不同的情况下进行配置。你也可以说 Spanner 可以自动优化，但在这种情况下，你如何证明自己的能力呢？
    1. 不分析访问模式、需求、关键字大小、QPS 等，可能会导致数据库性能不佳，难以满足系统的需求和挑战。
    2. 主键、索引、分片/分区键的选择应该根据具体情况进行考虑和平衡。例如，需要考虑查询频率、数据访问模式、数据大小和一致性要求，然后选择最适合的解决方案来满足业务需求。
    3. 不建议对所有表使用同一种数据库，因为不同的表可能具有不同的数据结构、数据访问模式、访问频率等，适合的数据库也可能不同。
    4. 认为数据库是无所不能的，不可能默认情况下丢失数据或读取过时的数据是不正确的。在实际应用中，考虑数据安全和一致性，以及如何防止数据丢失或损坏是很重要的。
